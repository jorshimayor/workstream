This is a static walkthrough of the system that exists now: external Flow auth, project guides, locked policy context, task queue, submission packets, evidence, automatic checker runs, audit records, and the current end states before human review.

Actors And System Boundaries

Project Manager Creates projects, guides, tasks, and repairs setup defects.
Worker Claims a ready task, submits work, and fixes worker-facing revisions.
Flow Auth Issues identity tokens. Workstream verifies tokens only.
FastAPI Backend Owns lifecycle rules, API contracts, and object-level visibility.
Checker Runner Runs registered checkers after submission finalization and records results.
Postgres Durable source for guides, tasks, submissions, checkers, and audit.

End-To-End Flow

1

Flow token enters API

The caller sends a Flow bearer token. Workstream verifies issuer, audience, expiry, signature, and roles.

ActorContext roles
2

Project guide and policy are prepared

The project owner provides open-ended guide material and business terms. Workstream runs guide sufficiency review, derives the submission artifact policy, then an admin or project_manager approves the internal policy bundle before tasks can run.

Project ProjectGuide v1 GuideSufficiencyReport artifact/checker/review/revision/payment
3

Guide activation locks contract

Activation validates a passing or acknowledged sufficiency report, immutable guide source snapshot, approved submission artifact policy, effective project submission artifact policy hash, project pre-submit checker compiled bundle hash, approved generated project post-submit checker policy with matching guide, source snapshot, effective project policy, and pre-submit checker provenance, review policy allowed decisions, revision states, and payment policy. Task readiness later validates that the task locks the applicable guide snapshot, effective project submission artifact policy hash, pre-submit checker bundle hash, and approved provenance-matched project post-submit checker policy reference.

status=active one active guide
4

Task is created and screened

The task locks the current guide and policy versions, then moves through screening before it can become available.

SCREENING READY locked versions=v1
5

Worker claims and starts

A qualified worker claims the ready task. Object-level visibility narrows after claim.

TaskAssignment CLAIMED IN_PROGRESS
6

Pre-submit intake checks run

Workstream runs pre-submit checks from the locked project pre-submit checker policy before creating a submission row. Preflight failures return PreSubmitCheckResponse details. Blocked submission-create attempts return pre_submission_checker_failed with pass/fail/warning details.

no submission yet no checker run yet blocking failures stop creation
7

Submission packet is created

After blocking pre-submit checks pass, Workstream creates the submission with summary, package reference, artifact hashes, evidence references, and attestation.

Submission v1 EvidenceItem artifact_hash_manifest
8

Submission is finalized

The server records finalization against the task's server-owned guide and policy context, finalizes evidence rows, and schedules the checker run.

finalized_at submission_finalized
9

Automatic checkers execute

Registered checkers run against the exact finalized submission, package hash, artifacts, and policy context.

CheckerRun CheckerResult attempt=1
10

Current system routes result

The checker gate sends passing work to review queue, worker-fixable failures to revision, and setup defects to internal repair.

review_pending needs_revision task_setup_blocked

Current End Of The Flow

review_pending

All blocking post-submit checker policy requirements pass. The task is ready for the Week 3 human review engine.

needs_revision

Worker-fixable checker failures are shown with safe feedback. This is not a human review decision.

task_setup_blocked

Internal setup defect. Project managers repair the task or guide context, then run trusted checker retry.

Durable Records Created Along The Way

Project + Guide Project, ProjectGuide, GuideSourceSnapshot bundle, SubmissionArtifactPolicy, effective project policy, project PreSubmitCheckerPolicy, post-submit checker policy, review policy, revision policy, payment policy.
Task Queue WorkstreamTask, locked guide snapshot, locked effective project submission artifact policy hash, locked pre-submit checker bundle hash, TaskAssignment, status transitions.
Submission Packet Submission version, package URI, package hash, artifact hash manifest, evidence rows.
Checker Trail CheckerRun, CheckerResult, routing recommendation, attempt/current-run flags.
Audit Events Task creation, status changes, submission finalization, checker gate start/pass/block.
Visibility Rules Project managers see project scope; workers see assigned scope; unassigned workers are blocked after claim.
Auth Boundary External Flow token claims become ActorContext. Workstream does not own passwords or primary auth sessions.
Test Database Guard Real API drills run against local async Postgres test databases only, unless a write-risk override is explicit.

Hard Invariants

  • Guide activation fails if the immutable guide source snapshot, guide sufficiency report, submission artifact policy, effective project submission artifact policy hash, project pre-submit checker compiled bundle hash, approved generated project post-submit checker policy with matching provenance, or required review, revision, and payment policy context is missing.
  • Pre-submit intake failures create no submission row or submission version.
  • Revision resubmission states are limited to needs_revision in v0.1.
  • Submission packet does not carry guide or policy versions from the worker.
  • Submission finalization records against the task's server-owned guide and policy context.
  • Evidence rows finalize with the submission.
  • Checker runs bind to submission version, package hash, and artifact hashes.
  • Duplicate checker results fail the real API drill.
  • Worker-facing review decisions stay accept, needs_revision, and reject.
Clean path: active guide -> ready task -> finalized submission -> checker pass -> review_pending.
Worker-fix path: checker failure -> needs_revision -> new submission version.
Setup-defect path: task_setup_blocked -> project manager repair -> trusted checker retry.
